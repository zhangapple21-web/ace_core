## 一、考古发现

### 1.1 发现过程

今天执行 ACE 自动考古主循环时，遇到一个导入错误：

```
ImportError: cannot import name 'Lexicon' from 'core.lexicon'
```

**表面原因**：`core/lexicon.py` 和 `core/lexicon/` 同名，Python 包优先于模块。

**深层原因**：这不是一个代码 bug，这是一次**结构演化中的过渡态断裂**。

系统正在从"单体词库模块"向"分层词库体系"演化，演化到一半停了 —— 分类体系独立成包了，但主类还在旧文件里，下游 import 也没更。

### 1.2 模式识别

顺着这个线索做了一次全系统命名冲突扫描，发现了**同一种演化模式反复出现**：

| 实体 | 现象 | 演化阶段 |
|------|------|---------|
| lexicon | 模块和包同名 | 拆分进行到一半 |
| SyncPlan | 3 个版本并存 | 契约独立了，实现端没跟上 |
| ArtifactDecision | 2 个版本并存 | 同上 |
| Experience / ExperienceDeposition | 两份完全相同的代码 | 文件搬家搬到一半 |
| BaseWorker | 2 个版本 | 旧的没删，新的已建 |

**共同模式**：

> **契约/分类/定义先独立出去了，但实现端和调用端还在引用旧版。**
>
> 就像搬家 —— 家具先搬过去了，人还在老房子里住，两边各有一套东西。

这不是一个孤立的 bug，这是**系统演化到当前阶段的结构性特征**。

## 二、演化阶段判定

### 2.1 系统所处阶段

根据这次考古，系统正处于：

```
阶段 A：单体混沌        所有定义混在一起
    ↓
阶段 B：契约独立中      契约先抽离成层，实现端还没统一引用  ← 我们在这里
    ↓
阶段 C：引用统一        所有模块都从契约层导入，不再自己定义
    ↓
阶段 D：唯一真相源      契约层成为定义的唯一来源
```

### 2.2 证据

支持"契约独立中"阶段判断的证据：

1. **`core/contracts/` 目录已建立** — 有 authority_contract、sync_contract、curator_decision_contract 等
2. **但实现端还在自己定义** — repository_curator.py 里还有自己的 ArtifactDecision、SyncPlan
3. **命名空间开始分层** — governance/ 下也有 contracts.py，说明契约概念在扩散
4. **存在多版本并存** — 同一个概念有 2-3 个实现版本，是典型的过渡态特征

## 三、治理协议层的诞生

### 3.1 为什么不是继续修代码

遇到 lexicon 同名问题时，第一反应是"改代码修一下"。

但修完之后意识到一个问题：

> 这不是最后一个同名问题。
> 以后还会有类同名、函数同名、配置同名、任务同名、知识同名……
> 每次都临时修吗？每次修法都不一样吗？每次都没人记录吗？

这就是为什么需要 **GOV-001** —— 不是为了解决这一个问题，而是为了建立一套**处理所有同类问题的统一流程**。

### 3.2 治理协议层的意义

治理协议层不是又加了一层架构。

治理协议层是**系统的"宪法性文件"** —— 规定系统如何处理冲突、如何准入知识、如何验证真相、如何演化结构。

有了治理协议层之后：
- 遇到问题 → 不是"谁看到谁顺手改了"
- 遇到问题 → "按 GOV-001 流程走，登记、分析、决策、沉淀"

这是系统从"开发工具"转向"长期运行的知识治理系统"的关键一步。
